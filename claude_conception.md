# 麻将牌系统构想（Claude 版）

> 目标：把现有「宝石 TD」的宝石体系整体替换为**麻将牌**体系。本文在现有引擎与玩法框架之上，略带自由地探索**多套玩法方案**（功能 / 升级 / 牌型定位 / 自带风格），并对每套方案**逐档标注改造成本与受影响测试**，供技术与玩法总监取舍拼装。
>
> 方向基线（已与总监对齐）：
> - **全谱系覆盖**：从「轻主题皮肤」到「高还原做牌」各出一套，方便对比。
> - **逐档标注成本**：每套方案标注改动量（最小 / 中 / 大）与受影响测试。
> - **三段为主 + 替代**：以「万条筒＝基础牌 / 中发白＝升级功能牌 / 东南西北＝终极牌」为主轴，另附替代定位。
>
> 优化优先级：**深度 > 清晰 > 手感**（沿用项目一贯取向；本文所有方案都要求「改玩法必补回归测试」）。

---

## 0. 现有系统与改造接缝（自足速览）

先把「必须尊重的事实」摆出来，后文所有方案都在这套接缝上落地。数值唯一事实源在 `src/game/config/`，纯逻辑在 `src/game/engine/`，React 编排在 `useGameEngine.ts`。

### 0.1 当前类型骨架

| 维度 | 现状 | 定义位置 |
|---|---|---|
| 基础"宝石" | 8 类（`GemType`：amethyst/diamond/topaz/opal/ruby/sapphire/emerald/obsidian） | `types/game.ts` |
| 品阶 | 4 档（`GemLevel`：chipped→flawed→normal→flawless，末档不可再合） | `types/game.ts` |
| 特殊塔 | 6 类（`SpecialTowerType`：silver/malachite/starRuby/moonstone/jade/onyx），均固定 `normal` | `types/game.ts` / `config/towers.ts` |
| 敌人 | 4 类（basic/fast/tank/boss） | `config/enemies.ts` |
| 波次 | 恰好 12 波，第 12 波为 boss 波，血量倍率 1.0→7.0 | `config/waves.ts` |

### 0.2 效果调色板（任何"牌"都要落到这套已测机制上）

塔身与子弹上都**扁平化携带**下列字段（`Tower`/`Bullet` in `types/game.ts`）：

- `damageType`：`physical`（护甲减免 `armor/(armor+10)`）/ `magic`（按 `magicResist` 比例减）/ `pure`（无视一切）。
- 效果标志：`multiTarget`、`splashRadius`（溅射，次要目标 0.5×且不暴击）、`slowEffect`（减速，封顶 90%、固定 3s）、`critChance`/`critMultiplier`（暴击，先于减免结算）、`pierce`（穿透，最近 60px 内 0.6×）、`poisonDamage`/`poisonDuration`（毒，可叠加、每秒一跳）、`stunChance`/`stunDuration`（眩晕，硬停、不可被推）。
- **无属性克制系统**：唯一的"克制"就是 `pure` 无视护甲/魔抗，因此 `ruby/starRuby/onyx` 是现成的坦克/boss 解。（这是一个可补深度的空位，见模块 §5.7。）

### 0.3 回合循环与经济不变量

```
building →(放满 3 座)→ deciding →(留 1 座)→ ready →(开波)→ playing
playing ─非最终波结算→ building（建造次数补回 3）
playing ─第 12 波结算→ victory
playing ─矿坑归零→ game_over
```

- **实为「三选一」**（`ECONOMY_CONFIG.towersPerRound = 3`，UI 文案即「三选一」）。每回合净得 **+1 保留塔 + 2 障碍**。
- **24 障碍不变量**（被测试锁死）：12 回合 × 2 障碍 = 恰好 `maxObstacles: 24`，正常流程永不触发提前"风化"。**任何改动"三选一"的 N，都会连带重算 `maxObstacles` 与 `building.ts` 容量逻辑。**
- 经济（`config/economy.ts`）：`startingWood 3 / woodPerRound 3 / towerWoodCost 1 / startingGold 50 / obstacleRemovalGoldCost 20 / maxObstacles 24 / startingMineHealth 15`。
- **金币只来自击杀**；无波末返利、无矿坑回血（`getStateAfterMineDamage` 明确拒绝负伤害/治疗）。
- `gameLevel` 用金币升（`100·2^(n-1)`），仅影响落塔的**品阶概率**（随机落塔永不出 flawless，末档只能靠合成）。

### 0.4 合成语法（当前是两段确定性合成）

- **升阶**：两座**同型同阶**基础塔 → 升一阶（纯函数 `findSynthesizableTowerPairs`）。
- **特殊**：两种**指定基础宝石** → 一座 `normal` 特殊塔（`SPECIAL_TOWER_RECIPES` 多重集匹配）。
- **随机只发生在落塔那一刻**（`placeTower` 内硬编码 8 宝石数组 @ `useGameEngine.ts:~277`），合成本身零随机。
- 结果落在被点击的**锚点塔**（保留 id 与格位）；被消耗塔的格子变为障碍。

### 0.5 关键改造接缝（决定成本的四处）

1. **类型**：`types/game.ts` 的三个枚举（改这里，TS 会把所有 `Record<...>` 需要更新处全部报出来）。
2. **配置**：`config/towers.ts` 的 `BASE_TOWER_STATS`（8×4）、`SPECIAL_TOWER_RECIPES`（6）、名称/颜色/图标表、`TOWER_LEVEL_PROBABILITIES`、`randomizeTowerLevel`、`calculateUpgradeCost`。
3. **匹配逻辑**：`engine/synthesis.ts` + `config/towers.ts` 的 pairs/recipes。
4. **随机 roll**：`useGameEngine.ts:~277` 的硬编码数组（**不**从 `GemType` 派生，必须同步改）。
5. **死字段可复活**：`GameState/EngineUiState.availableGems: GemType[]` 声明了但只赋 `[]`，是"摸 N 张牌"做牌流的现成接口。

### 0.6 测试骨架（会因身份/形状变化而红）

- 纯身份类（改名即需更新）：`config/towers.test.ts`、`rendering/spriteRegistry.test.ts`。
- 形状锁定类（改结构才需重设计）：`economy.test.ts`（"恰好三次一价落塔"）、`waves.test.ts`（"恰好十二波 + boss 收尾"）、`building.test.ts`（"12 个三塔回合无提前风化"）、`components/mobileLayout.test.tsx`（"五资源卡 / 三落塔"）。
- 逻辑类（新增玩法要补）：`synthesis.test.ts`、`combat.test.ts`、`enemyMovement.test.ts`、`gameFlow.test.ts`。

## 1. 麻将牌面总表与共用映射（所有方案的公共词汇）

### 1.1 牌面清单

| 类别 | 牌 | 张数 | 拟定层级定位 |
|---|---|---|---|
| 数牌 | 万 1–9、条 1–9、筒 1–9 | 27 面 | **基础牌**（工作马塔） |
| 三元（箭）牌 | 中 / 发 / 白 | 3 面 | **升级 / 功能牌**（催化与全局） |
| 风牌 | 东 / 南 / 西 / 北 | 4 面 | **终极牌**（役满级 archetype） |
| 花牌（可选） | 春夏秋冬 / 梅兰竹菊 | 8 面 | **遗物**（roguelite perk，见 §5.6） |

### 1.2 三花色 = 三门派（把 8 种效果干净地并入 3 花色）

这是全谱系方案共享的"门派身份"底座——三花色恰好可吸收现有全部 8 种效果原型：

| 花色 | 意象 | 门派定位 | 吸收的旧宝石身份 | 引擎字段落点 |
|---|---|---|---|---|
| **万**（万子·财） | 万＝大数、金光、钱 | **物理爆发·单体·财运** | amethyst(暴击)+ruby(纯粹爆)+diamond(部分) | `physical`/`pure`、`critChance`/`critMultiplier`、（可选）击杀多得金 |
| **条**（索子·箭竹） | 箭、雀鸟、藤蔓毒 | **连射·穿透·毒素·多目标** | diamond(多目标)+sapphire(穿透)+emerald(毒) | `magic`、`multiTarget`、`pierce`、`poisonDamage/Duration` |
| **筒**（筒子·铜钱/炮饼） | 圆、涟漪、炮、冰霜 | **范围·减速·眩晕·控场** | topaz(溅射)+opal(减速)+obsidian(眩晕) | `splashRadius`、`slowEffect`、`stunChance/Duration` |

> 设计价值：三花色天然对应「爆发 / 输出扩散 / 控场」三种 TD 支柱，玩家"清一色"某花色 = 押注某一支柱，取舍清晰。

### 1.3 数字 1–9 的两种用法（方案分野的关键）

- **作材料**（方案 B）：数字用于组"面子"——顺子要连续、刻子要相同——本身不定强度。
- **作战力档**（方案 D）：数字 1→9 即强度轴，取代 4 品阶。
- **幺九 vs 中张**：麻将区分 1/9（幺九，端牌）与 2–8（中张）。可让幺九塔走"极端型"（要么廉价工兵、要么稀有强端），中张走"通用主力"，为牌面增加纹理。

### 1.4 三元牌（中发白）= 功能催化 + 役满

| 牌 | 色/意象 | 功能定位 | 引擎实现 |
|---|---|---|---|
| **中**（红中） | 红＝火＝进攻 | 进攻催化：+伤害 / +暴击 / 附灼烧 | 提升 `damage`/`critChance`，或注入 `poison`（灼烧） |
| **发**（发财） | 绿＝财＝成长 | 经济催化：击杀多得金 / 降升级成本 / 波末返利 | 结算侧提升 `reward`/`gold` 收益（引擎小改） |
| **白**（白板） | 白＝空＝辅助 | 辅助催化：+射程 / +减速 / +眩晕 / 净化 | 提升 `range`/`slowEffect`/`stunChance` |

两种用法：①**贴牌催化**（消耗龙牌 + 基础塔 → 强化基础塔）；②**箭刻**（3 张同龙 → 全局光环），**大三元**（三箭刻齐）→ 役满级全场爆发。

### 1.5 风牌（东南西北）= 四终极 archetype + 圈风

| 风 | 五行/方位 | 终极定位 | 引擎实现 |
|---|---|---|---|
| **东** | 木·庄家 | **王牌单体**：极高纯粹伤害 + 高暴击，boss 杀手 | 高 `damage` + `pure` + `critChance` |
| **南** | 火 | **烈焰范围**：大溅射 + 持续灼烧毒 | 大 `splashRadius` + `poison` |
| **西** | 金 | **锐金破甲**：穿透 + 无视护甲 + 多目标 | `pierce` + `pure` + `multiTarget` |
| **北** | 水 | **寒水封锁**：强减速 + 高眩晕 + 范围控 | 高 `slowEffect` + `stunChance` + `splash` |

由**风刻**或指定面子配方合出；**大四喜**（四风齐）＝通关级役满追求。风牌另可驱动**圈风轮转**（§5.4）。

> **五行子主题（自带风格，可插拔）**：东木 / 南火 / 西金 / 北水 / 中土（中央戊己土，恰好对应"中"）。可据此做「敌人属性 × 塔属性」的相生相克层，填补 §0.2 的"无克制"空位（见 §5.7）。

## 2. 玩法方案（全谱系，逐档标注成本）

四套方案沿"麻将还原度 / 改造成本"由轻到重排列。A 为主轴轻皮肤、B 为主轴旗舰、C 为主轴重度做牌、D 为**替代定位**（数字即战力）。

---

### 方案 A —「熟张皮肤流」（轻主题·重手感）

**一句话**：机制骨架完全不动，只把宝石换成麻将皮肤 + 麻将术语的手感演出。**先拿到麻将壳，零风险上线。**

- **牌型定位**：沿用三段轴作**命名/占位**——8 基础塔套三花色命名，4 品阶换麻将品质名（如 生张→熟张→老张→绝张），6 特殊塔换 6 个番型名（银塔→"平胡"、星红宝石→"清一色"…）。中发白/东南西北暂作特殊塔高阶皮肤与未来占位，不落新机制。
- **功能 / 升级**：与现状一一对应，数值不动。
- **自带风格（纯表现层，几乎零逻辑改）**：
  - 合成成功弹「碰！/ 杠！/ 胡！」字样 + 音效；
  - 伤害飘字改用麻将术语配色；
  - 落塔 preview 复用现有"中性 marker"（本就不解析塔/障碍精灵，零改）。
- **改造成本：最小**。仅动 `types/game.ts`（重命名）、`config/towers.ts`（名称/颜色/图标表）、`spriteRegistry.ts` + 美术、`useGameEngine.ts:~277` 硬编码数组。合成/经济/波次/24 障碍不变量**全不动**。
- **受影响测试**：主要是身份字符串断言（`towers.test`/`spriteRegistry.test`）；逻辑测试基本不动。
- **优 / 劣**：✅ 上线快、风险极低、稳拿主题与更好手感。❌ 深度零提升，"麻将"仅停在皮肤，牌型/番种/做牌乐趣未兑现。
- **建议**：作为 **P0 铺垫**，为后续方案铺好美术与术语反馈层。

### 方案 B —「三花色·面子合成流」（★ 旗舰推荐）

**一句话**：把麻将的**面子语法**（吃/碰/杠/胡）变成**合成语法**，严格落地三段轴。深度与可行性的最佳平衡点。

#### 基础层（万条筒）
- 3 花色 = 3 门派（§1.2）。落塔随机 roll **花色 + 点数**（替换现有 8 宝石 roll）。
- 建议**保留 4 品阶轴**（换麻将品质名）以省测试：**点数定"是什么/材料身份"，品阶定"初始品质"**，二者正交。（若想让点数吸收品阶，见方案 D。）

#### 升级层 = 面子合成（核心乐趣：碰 vs 吃 的取舍）
| 面子 | 条件 | 产物 | 风格 |
|---|---|---|---|
| **刻子（碰）** | 3 张同花色同点数 | "明刻塔"：点数不变、单项大幅强化 | 走**单点爆发** |
| **顺子（吃）** | 同花色连续 3 点（如 三四五万） | "顺子塔"：链式/多目标/线形 | 走**扩散输出** |
| **杠** | 4 张同花色同点数 | "杠塔"：基础形态最强 | 稀有高投入 |
| **对子（将/眼）** | 2 张同 | 终极合成的"眼"材料，或临时小 buff | 铺垫 |

> 这与真麻将"碰走刻、吃走顺"的取舍同构：同一批牌，玩家在"单强"与"扩散"间做局部决策——这是 B 方案的深度内核。

#### 功能层（中发白）
- 摸到龙牌可：①**贴塔催化**（中→攻强 / 发→财 / 白→控辅，§1.4）；②凑**箭刻**→全局光环；③**大三元**→役满级全场爆发。

#### 终极层（东南西北）
- 4 终极塔（§1.5），由**风刻**或"指定面子配方"合出（例：东塔 = 一副清一色万顺 + 东风×N）。**大四喜**＝通关级追求。

#### 自带风格
- **番种被动**（§5.2）：全场牌面达成番型 → 被动加成（清一色→本派强化、碰碰胡→攻速、七对→暴击…），用纯函数 `evaluateBoardFan(towers)` 计算，战斗零侵入。
- **动词化合成 UI**（§5.1）：吃/碰/杠/胡 + 术语飘字。

- **改造成本：中**。重构 `BASE_TOWER_STATS`（3 花色 × 点数缩放，或 3×9）、扩 `synthesis.ts` 增顺子/刻子/杠匹配、加"风牌终极"枚举与配方（扩 `SpecialTowerType` 或新增 `UltimateType`）、改 `placeTower` roll。三选一保持 → **24 障碍不变量不动**；番种为纯函数附加。
- **受影响测试**：`towers.test`（配方/pairs）、`synthesis.test`（新增顺/刻/杠用例）、`config/towers` 结构；`mobileLayout` 若资源卡数不变则不动。**须为顺子/刻子/杠/番种各补回归测试**（符合"改玩法必补测试"约定）。
- **优 / 劣**：✅ 麻将味浓、深度大、手感强，吃碰取舍是核心；落在现有确定性合成框架内可行。❌ 合成 UI 与匹配逻辑要扩，测试补写量中等。
- **建议**：**P1（基础+功能层）→ P2（终极层+番种+圈风）** 分两步落地。

### 方案 C —「摸牌做牌·听牌役满流」（高还原·做牌为核）

**一句话**：把整局 12 波当作"做一手好牌"的过程——摸牌、打牌、听牌、自摸。**最高麻将还原度，最高深度上限。**

- **摸牌（复活 `availableGems`）**：每回合"摸" N 张候选牌，玩家从**手牌**中选牌落塔；可**弃牌**（打出 → 进牌河/变障碍）。把"随机落塔"升级为"抽牌—打牌"决策。
- **听牌 / 胡牌**：全场塔构成"牌面"，接近某番型时进入**听牌**（高亮 + 奖励预览）；达成 → **自摸**役满级奖励（全屏清怪 / 永久番 buff / 矿坑回血——注意这是**新机制**，现引擎明确禁止回血）。
- **番作 meta 货币/评分**：累计番数 = 分数，驱动结算奖励曲线、段位（役满/满贯/跳满）。
- **圈风/门风轮转**（§5.4）：12 波分 4 圈（东南西北场），当值风给对应门派/终极塔本圈加成。
- **宝牌 dora**（§5.5）+ **花牌遗物**（§5.6）：增加流派 variance 与 roguelite 深度。
- **改造成本：大**。复活并重做抽取/手牌 UI、经济与结算（番→奖励、可能加回血）、`gameFlow` 增听牌/自摸态、波次分圈。**24 障碍不变量、三选一、五资源卡、12 波** 多处测试需重设计。
- **受影响测试**：`economy.test`、`mobileLayout.test`、`gameFlow.test`、`waves.test` 均大改，需成体系补测试。
- **优 / 劣**：✅ 玩法最独特、深度上限最高、真正"打麻将"。❌ 工程量与风险最大，偏离现有精巧不变量。
- **建议**：作为 **P3 深度扩展包**，在 B 稳定后择优引入（可先只上"听牌反馈 + 番评分"这半套，不动摸牌）。

### 方案 D —「数字即战力·顺子流」（★ 替代定位）

**一句话**：替代主轴——**点数即战力档**，弱化 4 品阶，主打花色内 1→9 的成长与顺子推进。字牌**降为番役修饰**而非独立层级。

- **牌型定位（与三段轴的张力所在）**：塔强度主要由**点数 1–9** 决定，取代 chipped→flawless；9 点为该花色"满点"终极（类比 flawless）。
  - 落塔 roll 花色 + 低点数（高点看等级概率）；
  - **顺子**（连续 3 点）→ 推出更高点数段/顺子专属塔；**刻子**（3 同）→ 点数×档跃升。
- **字牌降级为"番役牌"**（更贴近真麻将"字牌凑番"）：
  - **白板 = 万能牌（百搭）**：可当任意花色/点数凑顺凑刻（§5.8）；
  - **红中/发财/风牌**：镶嵌后改变塔的**规则**（红中＝该塔全伤 +X%、发财＝该塔击杀返金、风牌＝赋予对应终极词条）。
- **改造成本：中—大**。把 `GemLevel` 4 档换成 1–9 点数轴 → `BASE_TOWER_STATS` 结构、`randomizeTowerLevel`、`calculateUpgradeCost`、`TOWER_LEVEL_PROBABILITIES` 均改；新增顺子/万能牌逻辑。
- **受影响测试**：`towers.test`（品阶/pairs 断言）、`synthesis.test`、`config` 概率表大改。
- **优 / 劣**：✅ 数值成长直观、顺子推进爽、"万能牌凑顺"很麻将。❌ 与用户三段定位有张力（字牌降为修饰）、品阶轴重构牵连测试较多。
- **建议**：作为 **B 的替代/备选主轴**并列评估；若偏好"数值成长"手感而非"番种收集"手感，则选 D。

## 4. 可插拔玩法模块（任意方案可叠加，独立标价）

这些是横切模块，能像"番役"一样组合进任一主轴，逐个独立评估。

| # | 模块 | 作用 | 引擎落点 | 成本 |
|---|---|---|---|---|
| 5.1 | **吃碰杠胡·动词化合成 UI** | 合成入口从"选一对"升级为麻将动词（碰=刻/吃=顺/杠=4/胡=终极）+ 术语飘字与音效 | `SynthesisDialog` + `synthesis.ts` 扩展 | 中 |
| 5.2 | **番种被动加成** | 清一色/混一色/碰碰胡/七对/大三元/大四喜 → 全场被动 | 纯函数 `evaluateBoardFan(towers)` + 展示，战斗零侵入 | 小—中 |
| 5.3 | **听牌 / 胡牌反馈** | 接近番型时高亮 + 奖励预览，极大提升目标感与手感 | UI + `evaluateBoardFan` 复用 | 中 |
| 5.4 | **圈风 / 门风轮转** | 12 波分 4 圈，当值风加成对应门派，给中后期节奏 | 波次元数据 + 一个加成乘子 | 小—中 |
| 5.5 | **宝牌 dora** | 每局随机"热门牌"全程加成，增 replay variance | roll 一次 + 加成乘子 | 小 |
| 5.6 | **花牌遗物**（春夏秋冬梅兰竹菊） | roguelite 被动 perk | 新增遗物系统与 UI | 中 |

## 5. 推荐路线与分期落地

**核心推荐：以方案 B 为骨架，分期推进，低成本模块优先叠加；C/D 作为深度扩展与替代备选。**

- **P0 — 皮肤与手感**（≈方案 A）：换牌面美术 + 术语反馈（碰/杠/胡飘字与音效）。零风险，先把麻将壳与手感层落地。
- **P1 — 三花色门派 + 面子合成**（B 基础层 + 功能层）：三花色门派、顺/刻/杠合成、中发白催化。**补齐顺子/刻子/杠/箭刻的回归测试。**
- **P2 — 风牌终极 + 番种 + 圈风**（B 终极层 + 模块 5.2 / 5.4 /（5.1、5.3））：四终极塔与大四喜、番种被动、圈风轮转、动词化合成与听牌反馈。
- **P3 — 深度扩展（择优）**：C 的摸牌/听牌/自摸、D 的点数轴、5.7 五行克制、5.6 花牌遗物。按玩家反馈与预算取舍。

**每阶段硬约束**：`npm run lint`／`npm test`／`npm run build` 三绿；保持 24 障碍不变量（或**显式重算并补测试**）；改任何玩法规则必附可复现回归测试（先加红、再转绿）。

## 6. 开放问题（落地前需总监拍板）

1. **三选一是否改 N？** 一旦改（如做牌流的手牌），需重算 `maxObstacles` 与 `building.ts` 容量，并重设 `economy.test`/`mobileLayout.test`。
2. **是否允许矿坑回血？** 现引擎明确禁止；役满/自摸奖励若含回血是**新机制**，需改 `getStateAfterMineDamage` 并补测试。
3. **点数轴 vs 4 品阶轴**：二选一（D）还是正交并存（B）？决定 `GemLevel` 是否重构。
4. **字牌定位**：独立层级（用户三段轴，A/B/C）还是番役修饰（D）？这是**主轴的最终取舍**。
5. **五行克制是否引入？** 会改变"纯粹伤害＝万能解"的现状，是加深度还是加复杂度需权衡。
6. **美术预算**：34 牌面 + 终极/番型/自摸演出的素材量；是否复用现有 `spriteRegistry` 一处映射（UI 与 canvas 共用）以摊薄成本。
7. **命名与国际化**：麻将术语（吃碰杠胡/番种）是否保留中文原味，还是给英文/图形化以适配更广受众。
